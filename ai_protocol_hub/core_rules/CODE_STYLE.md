# 代码规范

目标：保持代码一致性、可维护性、可读性

## 1. 缩进
- 使用 2 个空格缩进（禁止 Tab）
- 多行代码延续行缩进 2 个空格

## 2. 命名

| 类型 | 规范 | 示例 |
|------|------|------|
| 变量 | 驼峰命名法，有意义的名称 | `userName`, `isLoading` |
| 布尔值 | 使用 `is/has/can/should` 前缀 | `isValid`, `hasPermission` |
| 函数 | 动词开头 | `openFile()`, `handleClick()` |
| 常量 | 全大写 + 下划线 | `MAX_SIZE`, `API_URL` |
| 组件/模块 | 名词形式 | `UserManager`, `DataParser` |

## 3. 代码组织
- 新功能独立模块化，避免全部写在 app.js
- 保持模块代码行数适中
- 每个文件必须有功能描述注释

## 4. 项目名称
- 展示时统一使用项目代号，无代号则用项目名称

## 5. 注释

### 5.1 基本要求
- 必须使用中文注释
- 注释与代码同步更新
- 空行不超过 1 行
- 文件头部包含：职责简介、功能、模块索引

### 5.2 函数/方法注释
- 每个函数必须有注释
- 包含：功能描述、参数说明、返回值说明
- 复杂函数说明实现思路

### 5.3 类/模块注释
- 每个类/模块必须有注释
- 包含：功能描述、主要属性、核心方法

### 5.4 特殊代码注释
- 易错代码说明原因和避免方法
- 临时方案标记后续处理方向

## 6. 代码生成改动

### 6.1 修改代码
- 必须先阅读注释了解原有逻辑

### 6.2 新功能实现
- 考虑可维护性和扩展性
- 规划好目录结构
- 脚本注意中文编码
- 优先复用现有技术和方案
- 使用新技术后总结文档（技术名称、场景、API、示例、注意事项）
- 保持代码简洁，避免冗余
